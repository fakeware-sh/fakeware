This section is for people who want to work **on** Fakeware itself. If you're here to seed a shop, start with the [Usage](/docs/usage/guide/introduction) section; if you want to extend Fakeware without forking it, see [Plugin development](/docs/development/plugins/overview).

## The monorepo

Fakeware is a [Bun](https://bun.sh) workspaces monorepo. Three workspace globs (`apps/*`, `packages/*`, and `plugins/*`) hold everything:

```bash
fakeware/
├── apps/
│   ├── cli/                 # @fakeware/cli: the 'fakeware' command
│   └── create-fakeware/     # create-fakeware: the 'npm create fakeware' scaffolder
├── packages/
│   └── core/                # @fakeware/core: the library the CLI is built on
├── plugins/
│   └── plugin-pickware/     # @fakeware/plugin-pickware: reference plugin
├── package.json             # workspace root + scripts
└── biome.json               # lint + format config
```

## The packages

::field-group
  :::field{name="@fakeware/core" type="packages/core"}
  The library. Contains the `define` / `ref` / `shop` authoring language, the resolution engine (`up` / `down` and the manifest), the Shopware client and shop-context fetchers, and the plugin system.
  :::

  :::field{name="@fakeware/cli" type="apps/cli"}
  The `fakeware` binary (`init`, `up`, `down`). A thin [commander](https://github.com/tj/commander.js) wrapper over `@fakeware/core`.
  :::

  :::field{name="create-fakeware" type="apps/create-fakeware"}
  The `npm create fakeware` scaffolder. Delegates to the CLI's `init` logic.
  :::

  :::field{name="@fakeware/plugin-pickware" type="plugins/plugin-pickware"}
  A real plugin that seeds Pickware ERP data. The reference implementation for [plugin development](/docs/development/plugins/overview).
  :::
::

## `@fakeware/core` subpaths

The core package exposes four entry points. Knowing which is which is the key to reading (and extending) the codebase:

::field-group
  :::field{name="@fakeware/core"}
  The authoring language and public runtime: `define`, `many`, `keyed`, `ref`, `shop`, `shopToken`, `builders`, `assocIds`, `definePlugin`, `readManifest`, `runUp` / `runDown`, and the error types.
  :::

  :::field{name="@fakeware/core/config"}
  `defineConfig` and the config types.
  :::

  :::field{name="@fakeware/core/shopware"}
  The lower-level Shopware layer: `createShopwareClient`, `createSyncSink`, `validateConnection`, `withRetry`, `price`, `shop`, and the shop-context types.
  :::

  :::field{name="@fakeware/core/testing"}
  Test helpers for plugin authors: `createTestPluginContext`, `runPluginHookOnce`, `createCollectingLogSink`.
  :::
::

## How the CLI wires to core

The CLI owns no business logic. Each command loads config and hands off to a `runUp` / `runDown` entry point in core:

```ts
// apps/cli/src/commands/up.ts (shape)
import { runUp } from '@fakeware/core'
import { loadConfig } from '@fakeware/core/config'
import { createShopwareClient, createSyncSink } from '@fakeware/core/shopware'

const loaded = await loadConfig({ config: opts.config })
const client = createShopwareClient(loaded.connection)
const sink = createSyncSink(loaded.connection, { client })

await runUp({ loaded, client, sink, dryRun: opts.dryRun, reporter, fakewareVersion })
```

`down` mirrors this with `runDown`. The CLI's only extra job is the **reporter**, which bridges core's `Reporter` interface to [`@clack/prompts`](https://github.com/bombshell-dev/clack) spinners. This separation is why the plugin system and testing kit can exercise the whole engine without the CLI.
