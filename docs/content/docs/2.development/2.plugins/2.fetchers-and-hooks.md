## Fetchers

A fetcher adds data to the **shop context**, the snapshot Fakeware builds at the start of a run. Your fetched data lands in `context.extensions`, where definitions read it via `shop.extensions` or a `shopToken`.

A `ShopContextFetcher` has three members:

```ts
import type { ShopContextFetcher } from '@fakeware/core'

export const warehousesFetcher: ShopContextFetcher = {
  entity: 'pickware warehouses',
  fetch: async (client) => {
    // client is the authenticated Admin API client
    const raw = await client.invoke('searchPickwareWarehouse post /search/pickware-erp-warehouse', {
      body: { page: 1, limit: 500 },
    })
    return raw
  },
  merge: (data, raw) => {
    data.extensions.pickwareWarehouses = rowsOf(raw)
  },
}
```

::field-group
  :::field{name="entity" type="string"}
  A label for the data, used in progress and error messages (`Plugin "x" fetcher "entity" failed.`).
  :::

  :::field{name="fetch(client)" type="=&gt; Promise&lt;unknown&gt;"}
  Called with the Shopware client. Return whatever raw data you need; paginate here if the result is large.
  :::

  :::field{name="merge(data, result)" type="=&gt; void"}
  Write your data into `data.extensions[...]`. Runs after `fetch` resolves.
  :::
::

To consume fetched data during resolution, pair it with a `shopToken`, a deferred lookup that resolves against the live context:

```ts
import { shopToken } from '@fakeware/core'

// resolves to a warehouse ID at build time
warehouseId: shopToken('pickwareWarehouse:main', (s) => warehouseIdByCode(s.extensions, 'main'))
```

## Lifecycle hooks

Hooks fire at fixed points in a run. All are optional; implement only the ones you need. Each receives a context object and may be `async`.

::field-group
  :::field{name="configResolved(ctx)" type="ConfigContext"}
  First step of **both** `up` and `down`, before any shop I/O. Good for validating config.
  :::

  :::field{name="contextReady(ctx)" type="PluginContext"}
  After the shop context (including your fetchers) is fetched. Always runs on `up`; on `down` only if a plugin declares `contextReady`, `beforeRevert`, or `afterRevert`.
  :::

  :::field{name="beforeApply(ctx)" type="ApplyContext"}
  `up` only, right before the write plan is applied. `ctx.dryRun` tells you whether writes will happen.
  :::

  :::field{name="afterApply(ctx)" type="ApplyContext &amp; { result: UpResult }"}
  `up` only, after the plan is applied. `ctx.result` holds what changed.
  :::

  :::field{name="beforeRevert(ctx)" type="RevertContext"}
  `down` only, before manifest entities are deleted.
  :::

  :::field{name="afterRevert(ctx)" type="RevertContext &amp; { result: DownResult }"}
  `down` only, after the revert.
  :::

  :::field{name="onError(ctx)" type="ErrorContext"}
  Dispatched to **every** plugin when any hook throws. `ctx.error` and `ctx.phase` describe the failure. Errors thrown here are swallowed.
  :::
::

The context objects nest: `ConfigContext` (`config`, `connection`, `projectRoot`, `mode`, `logger`) is the base; `PluginContext` adds `shopContext`; the apply/revert contexts add `dryRun`.

```ts
export function myPlugin(): FakewarePlugin {
  return definePlugin({
    name: 'my-plugin',
    hooks: {
      configResolved({ config, logger }) {
        logger.info('config looks good')
      },
      afterApply({ result, logger }) {
        logger.info(`committed ${result.committed} change(s)`)
      },
    },
  })
}
```

::note
Hooks run **in registration order, sequentially**. If a hook throws, Fakeware dispatches `onError` to all plugins and then rethrows the failure wrapped in a `PluginError` (`Plugin "name" failed during "phase".`).
::

Use the injected `logger` (`debug` / `info` / `warn` / `error`) rather than `console`; it routes into the CLI reporter, and `debug` is suppressed unless enabled.
