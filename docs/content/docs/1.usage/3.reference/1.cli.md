The `fakeware` binary has three commands: `init`, `up`, and `down`. This page documents each one exactly.

## Global options

Available on the root command:

::field-group
  :::field{name="-V, --version"}
  Print the version and exit.
  :::

  :::field{name="--no-color"}
  Disable ANSI colors in output. Also honored via the `NO_COLOR` environment variable.
  :::

  :::field{name="-h, --help"}
  Show help. Available on every command.
  :::
::

```bash [Terminal]
fakeware --help
fakeware up --help
```

## `fakeware init`

Scaffold a project: a `package.json`, a typed `fakeware.config.ts`, and an `.env`, then install the config types. This is the same routine `create-fakeware` runs.

```bash [Terminal]
fakeware init [options]
```

### Options

::field-group
  :::field{name="--url &lt;url&gt;"}
  Shopware URL to pre-fill the connection.
  :::

  :::field{name="--client-id &lt;id&gt;"}
  OAuth2 client ID.
  :::

  :::field{name="--client-secret &lt;secret&gt;"}
  OAuth2 client secret.
  :::

  :::field{name="--output &lt;path&gt;"}
  Directory to scaffold into. Defaults to the current directory.
  :::

  :::field{name="--secrets &lt;dest&gt;" type="env | inline"}
  Where to put credentials: a git-ignored `.env` (`env`, the default) or written directly into the config (`inline`).
  :::

  :::field{name="--package-manager &lt;pm&gt;" type="bun | npm | pnpm | yarn"}
  Which package manager to use. Auto-detected if omitted.
  :::

  :::field{name="--no-install"}
  Write files but skip installing dependencies.
  :::

  :::field{name="--force"}
  Overwrite existing files in a non-empty directory.
  :::

  :::field{name="--dry-run"}
  Preview the files that would be written without writing them.
  :::

  :::field{name="--yes"}
  Accept defaults and never prompt. Use for non-interactive runs.
  :::
::

### Interactive vs. non-interactive

`init` is interactive by default, walking you through location, package manager, and optionally connecting to your shop (validating the connection before writing).

It runs **non-interactively** when the terminal isn't a TTY, when you pass `--yes`, or when you supply `--url`, `--client-id` and `--client-secret` together. In that mode it uses defaults and prompts for nothing, which is ideal for CI.

```bash [Terminal]
fakeware init \
  --url https://my-shop.example.com \
  --client-id "$SHOPWARE_CLIENT_ID" \
  --client-secret "$SHOPWARE_CLIENT_SECRET" \
  --yes
```

## `fakeware up`

Apply your data definitions to the shop.

```bash [Terminal]
fakeware up [options]
```

### Options

::field-group
  :::field{name="--config &lt;path&gt;"}
  Path to `fakeware.config.ts`. Defaults to searching the current directory and its parents.
  :::

  :::field{name="--dry-run"}
  Resolve and diff everything, report what would change, but write nothing and don't touch the manifest.
  :::
::

### Output

`up` reports per-entity progress with created / updated / unchanged counts, then a summary:

```text
Committed 210 changes to https://my-shop.example.com
```

If nothing changed, it prints `Already up to date. Nothing changed.` A dry run ends with `Dry run complete. Nothing was written.`

## `fakeware down`

Delete the demo data Fakeware created, per its manifest.

```bash [Terminal]
fakeware down [options]
```

### Options

::field-group
  :::field{name="--config &lt;path&gt;"}
  Path to `fakeware.config.ts`. Same resolution as `up`.
  :::

  :::field{name="--yes"}
  Skip the confirmation prompt. Required for non-interactive teardown.
  :::
::

### Behavior

`down` first reads the manifest for the configured shop. If there's no manifest, it's a friendly no-op (`Nothing to revert`). Otherwise it shows how many records will be removed and, unless `--yes` is given, asks you to confirm.

It deletes in reverse dependency order across multiple passes to tolerate Shopware delete conflicts (see [Up and down](/docs/usage/concepts/up-and-down)). On success:

```text
Removed 210 records from https://my-shop.example.com
```

### Exit codes

::field-group
  :::field{name="0"}
  All recorded records were removed (or there was nothing to remove, or the user aborted the prompt).
  :::

  :::field{name="1"}
  Some records could not be removed. The failed entities are named and kept in the manifest; run `down` again once the blocker clears.
  :::
::

## Configuration resolution

Both `up` and `down` locate the config the same way: use `--config` if given, otherwise search the current directory and walk up the tree for `fakeware.config.ts`. The directory containing the config is the project root, which anchors `data/` and `.fakeware/`. Details are in the [Configuration reference](/docs/usage/reference/configuration).
