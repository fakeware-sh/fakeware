Configuration lives in `fakeware.config.ts` at your project root. It's a real TypeScript module that exports a call to `defineConfig`.

## `defineConfig`

Import `defineConfig` from the `/config` subpath and export its result as the default export:

```ts [fakeware.config.ts]
import { defineConfig } from '@fakeware/core/config'

export default defineConfig({
  shopware: {
    url: '$SHOPWARE_URL',
    clientId: '$SHOPWARE_CLIENT_ID',
    clientSecret: '$SHOPWARE_CLIENT_SECRET',
  },
})
```

`defineConfig` is an identity helper. It exists purely to give you full type-checking and autocompletion on the config shape. It accepts either a config object or a function (see [Environment-aware config](#environment-aware-config)).

## The `shopware` connection

The one required section. It tells Fakeware how to reach and authenticate against your shop.

::field-group
  :::field{name="url" type="string" required}
  The base URL of your Shopware shop, e.g. `https://my-shop.example.com`.
  :::

  :::field{name="clientId" type="string" required}
  The OAuth2 client ID of a Shopware integration with admin access.
  :::

  :::field{name="clientSecret" type="string" required}
  The integration's client secret.
  :::
::

All three are required for `up` and `down`. Create the integration in the Shopware admin under **Settings → System → Integrations** (see [Installation](/docs/usage/guide/installation#connect-to-a-shop)).

::warning
Keep secrets out of tracked code. Reference them with the `$NAME` syntax and put the real values in a git-ignored `.env`.
::

## Environment variables

Any string value in the config that is **exactly** `$NAME` is replaced with the value of that environment variable at load time.

```ts
url: '$SHOPWARE_URL'          // replaced with process.env.SHOPWARE_URL (or .env)
```

The rules are precise:

::field-group
  :::field{name="Whole-string only"}
  The value must be exactly `$NAME`, meaning an uppercase/digit/underscore name after the `$`. Embedded forms like `https://$HOST/api` are left as-is, not interpolated.
  :::

  :::field{name="Missing is an error"}
  If the config references `$NAME` and it isn't set anywhere, Fakeware fails with a clear message pointing you at your `.env`.
  :::

  :::field{name=".env is read automatically"}
  Fakeware reads a `.env` file from the project root itself and merges it over `process.env`.
  :::
::

### `.env` format

A standard key-value file. Blank lines and `#` comments are ignored; surrounding single or double quotes on values are stripped.

```bash [.env]
SHOPWARE_URL=https://my-shop.example.com
SHOPWARE_CLIENT_ID=your-client-id
SHOPWARE_CLIENT_SECRET="your-client-secret"
```

## Environment-aware config

For values that depend on the environment, pass a **function** to `defineConfig`. It receives `{ env, mode }` and returns the config:

```ts [fakeware.config.ts]
import { defineConfig } from '@fakeware/core/config'

export default defineConfig(({ env, mode }) => ({
  shopware: {
    url: mode === 'production' ? env.PROD_URL : env.STAGING_URL,
    clientId: env.SHOPWARE_CLIENT_ID,
    clientSecret: env.SHOPWARE_CLIENT_SECRET,
  },
}))
```

::field-group
  :::field{name="env" type="Record&lt;string, string | undefined&gt;"}
  The merged environment: `process.env` with `.env` layered on top.
  :::

  :::field{name="mode" type="string"}
  The run mode, `development` by default.
  :::
::

`$NAME` interpolation still runs on the returned object, so you can mix both styles.

## Config resolution

When you run `up` or `down`, Fakeware finds the config like this:

1. If you passed `--config <path>`, it uses that (resolved relative to the current directory if not absolute).
2. Otherwise it looks for `fakeware.config.ts` in the current directory and walks **up** the tree until it finds one.

The directory containing the config becomes the **project root**, which anchors `data/` discovery and the `.fakeware/` manifest directory. If no config is found, Fakeware tells you to run `fakeware init` first.

## Plugins

The config accepts a `plugins` array to extend Fakeware with custom entities, shop-context fetchers, and lifecycle hooks:

```ts
import { pickware } from '@fakeware/plugin-pickware'

export default defineConfig({
  shopware: { /* … */ },
  plugins: [pickware()],
})
```

See [Plugin development](/docs/development/plugins/overview) for authoring your own, and the list of lifecycle hooks.
