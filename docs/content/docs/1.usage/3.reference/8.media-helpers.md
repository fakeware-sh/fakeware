Helpers for seeding images and product covers.

```ts
import { media } from '@fakeware/core' // describes one image; use it inline on a product
```

## `media(input)`

Describes one `media` entity record and where its file comes from. Returns a record you pass to `define('media', …)`.

::field-group
  :::field{name="url" type="string"}
  A remote URL. Shopware downloads it server-side. Provide **either** `url` or `file`, not both.
  :::

  :::field{name="file" type="string"}
  A local file path (absolute, or relative to the project root). Its bytes are uploaded. Provide **either** `url` or `file`.
  :::

  :::field{name="extension" type="string"}
  Optional. The file extension (e.g. `png`). Inferred from the `url`/`file` when omitted; **required** if it can't be inferred.
  :::

  :::field{name="alt" type="string"}
  Optional. Alt text.
  :::

  :::field{name="title" type="string"}
  Optional. Media title.
  :::

  :::field{name="fileName" type="string"}
  Optional. Base name for the stored file. A deterministic suffix is always appended so runs stay collision-free; defaults to the file's base name.
  :::

  :::field{name="private" type="boolean"}
  Optional. Marks the media private.
  :::

  :::field{name="mediaFolderId" type="string | ShopToken"}
  Optional. Defaults to `shop.mediaFolder('product')` (the Product Media folder).
  :::

  :::field{name="$key" type="string"}
  Optional. A key so you can `ref('media').key(...)` this image.
  :::
::

```ts
// inline on a product — hoisted into the media entity for you:
define('product', {
  $key: 'hoodie',
  cover: media({ url: 'https://picsum.photos/seed/1/800.jpg', alt: 'Hero' }),
})

// or as a shared, referenceable media entity:
define('media', [media({ $key: 'logo', file: './assets/logo.png' })])
```

## `cover` and `gallery` fields

Inline product fields, not function calls. `cover` holds one `media()` literal (or a media ref); `gallery` holds an array of them. Fakeware hoists each literal into a `media` row, uploads its bytes, and emits the Shopware `media` (`ProductMedia[]`) association and `coverId`.

::field-group
  :::field{name="cover" type="MediaRecord | ref token"}
  The main image. Becomes `product_media` position 0 and the `coverId`.
  :::

  :::field{name="gallery" type="Array<MediaRecord | ref token>"}
  Extra images, in order, appended after the cover. If `cover` is omitted, the first `gallery` entry becomes the cover.
  :::
::

The final Shopware gallery is **cover first, then gallery entries**; `coverId` points at the cover's `product_media` row (never a media id).

```ts
define('product', {
  $key: 'hoodie',
  cover: media({ file: './assets/hoodie.png', alt: 'Hoodie' }),
  gallery: [
    media({ url: 'https://picsum.photos/seed/back/800/800.jpg', alt: 'Back' }),
    ref('media').key('brand-logo'), // refs and literals may mix
  ],
})
```

### Deterministic ids

Each hoisted `media` row and its `product_media` association get a deterministic id derived from the **owning record and slot** — `media` `product:hoodie:cover`, `product:hoodie:0`, … — so runs are idempotent. Pass an explicit `$key` on a `media()` literal to override the derived id (and to share one image across records).

## How the upload works

Fakeware creates the `media` rows through the normal Sync API, then uploads each row's bytes with a single call per image to Shopware's media upload endpoint:

- **Endpoint:** `POST /api/_action/media/{mediaId}/upload`
- **Query:** `extension` (required), `fileName` (deterministic, sent automatically)
- **URL source:** `Content-Type: application/json`, body `{ "url": "…" }`; Shopware downloads the file.
- **File source:** `Content-Type: application/octet-stream`, body = the raw bytes.
- **Response:** `204 No Content`.

Uploads run inside the same transactional step as the `media` write, are retried on transient (429 / 5xx / timeout) failures, and surface as a `ShopwareApiError` otherwise (for example `CONTENT__MEDIA_DUPLICATED_FILE_NAME` if a file name collides). Because Fakeware only re-uploads media whose record hash changed, and re-uploading to the same media id replaces the file, repeated `up` runs are idempotent.
