Fakeware surfaces specific error types rather than generic failures, so you can tell configuration mistakes from shop rejections from data problems at a glance. This page lists each one.

## `ConfigError`

Something is wrong with your configuration or manifest.

**Common causes:**

- No `fakeware.config.ts` found in the current directory or any parent → run `fakeware init` first, or pass `--config`.
- The config doesn't `export default defineConfig(...)`.
- No `shopware` connection configured, so `up`/`down` have no shop to talk to.
- A referenced `$NAME` environment variable isn't set → check your `.env`.
- A manifest file is corrupt (checksum mismatch) or uses an unsupported version.

**Resolve:** fix the config or environment as the message indicates. For manifest version errors, upgrade Fakeware; for corruption, investigate the `.fakeware/` file.

## `RefError`

A reference couldn't be resolved during the build-plan stage.

**Common causes:**

- `ref('entity')` names an entity that was never defined.
- `ref('tax').key('standard')` names a `$key` that doesn't exist.
- `ref('product').at(n)` is out of range for the batch.
- `ref('entity').pick()` has no records to choose from.

**Resolve:** the message names the entity and what you asked for. Check the `$key` spelling, ensure the referenced entity is defined somewhere in `data/`, and confirm the index is in range.

## `ShopContextError`

A `shop.*` lookup didn't match anything in the live shop, or the shop returned no data for a default.

**Common causes:**

- `shop.country('XX')`, `shop.salutation('…')`, `shop.tax(rate)` etc. with a value the shop doesn't have.
- A default requested (`shop.defaultCountry`) but the shop returned none of that kind.
- Accessing `shop.context()` / `shop.extensions` outside of definition resolution.

**Resolve:** the message lists what *is* available (e.g. `Available: DE, AT, CH`). Correct the key, or pick a value the shop actually has.

## `GraphError`

The reference graph contains a **cycle** (entity A depends on B which depends, directly or transitively, back on A), so Fakeware can't pick a safe write order.

**Resolve:** break the cycle by restructuring the data. Often one side of a mutual relationship can be set in a separate follow-up entity, or expressed as a self-reference within a single entity instead of across two.

## `ShopwareApiError`

The Shopware API rejected a write or delete. This is the richest error type, carrying the shop's own response.

**Fields:**

::field-group
  :::field{name="status" type="number | null"}
  The HTTP status, e.g. `409`, `403`, `500`.
  :::

  :::field{name="entity" type="string | null"}
  The entity being written when it failed.
  :::

  :::field{name="errors" type="ParsedApiError[]"}
  The structured errors Shopware returned, each with `code`, `detail`, `field`, `pointer`, `recordId`.
  :::

  :::field{name="retryable" type="boolean"}
  Whether Fakeware considers the failure worth retrying (used by the converging teardown).
  :::
::

**Common causes:**

- A field value the shop's validation rejects.
- A delete blocked because another record still references the target (`409`). This is handled automatically during `down` by retrying across passes.
- A permissions problem (`403`) where the integration lacks the needed rights.
- A batch over the 5 MB per-entity request limit; reduce the number of records for that entity.

**Resolve:** read the `errors` detail and pointer to find the offending field or record. For permission errors, widen the integration's privileges in the admin.

## `ApplyStopped`

Raised when `up` halts because a write failed. It's the signal that the run stopped early; the underlying reason is the `ShopwareApiError` reported just before it.

**Resolve:** fix the reported Shopware error, then re-run `up`. Entities that committed before the stop remain in the manifest, so the re-run resumes cleanly and `down` can still clean up the partial state.

## `ShopwareConnectionError`

Fakeware couldn't reach or authenticate with the shop.

**Common causes:**

- Wrong URL, shop unreachable, or DNS/network failure.
- Invalid client ID or secret.
- The shop returned an unexpected response shape for a context query.

**Resolve:** verify the URL and credentials (`fakeware init` validates them), and confirm the shop is up. For a local dockware container, check the port mapping.

## `LoadModuleError`

Fakeware failed to load a TypeScript module, either your config or a `data/` file.

**Common causes:**

- A syntax or import error in the file.
- A path that doesn't exist.

**Resolve:** the original error message is included. Fix the offending file; run your type-checker to catch issues before applying.
