This quickstart takes you from nothing to real records in your shop. It assumes you have a Shopware shop and integration credentials ready. If not, start with [Installation](/docs/usage/guide/installation).

::steps{level="3"}

### Scaffold and connect

Create a project, then add your shop URL and integration credentials:

::code-group
```bash [npm]
npm create fakeware@latest my-seed
cd my-seed
```

```bash [pnpm]
pnpm create fakeware my-seed
cd my-seed
```

```bash [yarn]
yarn create fakeware my-seed
cd my-seed
```

```bash [bun]
bun create fakeware my-seed
cd my-seed
```
::

```bash [.env]
SHOPWARE_URL=https://my-shop.example.com
SHOPWARE_CLIENT_ID=your-client-id
SHOPWARE_CLIENT_SECRET=your-client-secret
```

The generated `fakeware.config.ts` already references those variables. See [Installation → Connect to a shop](/docs/usage/guide/installation#connect-to-a-shop) if you need to create an integration first.

### Write a data file

Fakeware evaluates every `.ts` file in the `data/` directory. Create one that defines a tax rate and a handful of products:

```ts [data/catalog.ts]
import { define, many, ref, shop } from '@fakeware/core'

define('tax', {
  $key: 'standard',
  name: 'Standard rate',
  taxRate: 19,
})

define(
  'product',
  many(10, (ctx) => ({
    name: `Demo product ${ctx.index + 1}`,
    productNumber: `SW-${10000 + ctx.index}`,
    stock: 100,
    active: true,
    taxId: ref('tax').key('standard'),
    price: [{ currencyId: shop.defaultCurrency, gross: 19.99, net: 16.8, linked: true }],
  })),
)
```

Two things to notice:

- `ref('tax').key('standard')` points the products at the tax record by its `$key`. Fakeware writes the tax **before** the products automatically.
- `shop.defaultCurrency` resolves to the shop's default currency ID at apply time.

### Preview the changes

Before writing anything, see what `up` would do:

```bash [Terminal]
fakeware up --dry-run
```

Fakeware connects, resolves your data, and reports what it would create, all without touching the shop.

### Apply

Now write it for real:

```bash [Terminal]
fakeware up
```

Fakeware syncs the tax and products and records them in a per-shop manifest. Open your admin and the products are there.

### Re-run and revert

Run `up` again and nothing changes. Fakeware only writes what actually drifted, so a repeat run is a no-op (`Already up to date. Nothing changed.`). This is [idempotency](/docs/usage/concepts/determinism) at work.

```bash [Terminal]
fakeware up
```

When you're finished, remove exactly what Fakeware created:

```bash [Terminal]
fakeware down
```

::

## What just happened

::card-group
  :::card{icon="i-lucide-file-code" title="You declared data" to="/docs/usage/concepts/defining-data"}
  `define()` calls in `data/*.ts` describe Shopware entities as plain objects.
  :::

  :::card{icon="i-lucide-workflow" title="Fakeware ordered it" to="/docs/usage/concepts/references"}
  References resolved to real IDs, and dependencies were written first.
  :::

  :::card{icon="i-lucide-fingerprint" title="It stayed idempotent" to="/docs/usage/concepts/determinism"}
  Deterministic IDs and hashing made the second run a no-op.
  :::

  :::card{icon="i-lucide-arrow-down-up" title="It's reversible" to="/docs/usage/concepts/the-manifest"}
  A manifest tracked every record so `down` could remove them exactly.
  :::
::
