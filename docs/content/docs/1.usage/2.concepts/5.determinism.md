Determinism is the property that makes Fakeware safe to re-run. The same definitions always produce the same records, so applying twice changes nothing and teardown is exact. Four mechanisms deliver it.

## Stable IDs

Every record gets its Shopware ID from a deterministic function of its **entity** and its **key**.

```text
id = uuidv5("entity:key")
```

- If a record has a `$key`, that's the key.
- If it doesn't, its position in the batch (its index) is used.

Because the ID is derived, not random, the record for `tax` / `standard` has the *same* UUID on every run, on every machine. That is what lets Fakeware recognize its own records and update them in place instead of creating duplicates.

::warning
The ID depends on the `$key` (or index). Rename a `$key`, and you get a different UUID. Fakeware then treats it as a new record and leaves the old one behind. Keep keys and index-derived fields stable across runs.
::

## Seeded generation

Each generated record carries a **seed** derived from its entity and index. Fakeware uses that seed to make random operations reproducible, most visibly `ref(...).pick()`:

```ts
categories: ref('category').pick(2)
```

The same product always picks the same two categories, because the pick is driven by the record's seed, not by unseeded chance. The seed is also exposed as `ctx.seed` if you want your own stable randomness.

::tip
Prefer `ctx.index`-derived values and seeded picks for anything that should stay stable. A bare `Math.random()` or `new Date()` produces a different value each run, so those records will always look "changed" and be rewritten every time.
::

## Content hashing

To decide whether a record actually changed, Fakeware computes a **content hash** of each resolved record. The record is first put into a canonical form (object keys sorted, structure normalized) and then hashed. On the next run, Fakeware compares the new hash to the one stored in the manifest:

- Same hash → **unchanged**, skipped entirely.
- Different hash → **updated**, rewritten.
- No prior hash → **created**.

This per-record diff is why a repeat `up` is a no-op, and why editing one product rewrites only that product.

## Invariance to shop IDs

Here's the subtle part that makes definitions portable. A record often contains shop-resolved IDs, such as a currency ID or a tax ID, and those differ from shop to shop. If the hash included the concrete IDs, moving to a shop with different internal IDs would make every record look changed.

Fakeware avoids this by hashing shop lookups by their **meaning**, not their resolved value. In the canonical form used for hashing, `shop.defaultCurrency` becomes a stable descriptor like `__shop__:defaultCurrency` instead of the actual UUID.

The result: the same definitions produce the same hashes against any shop, even though the real IDs written to each shop differ. Your data stays stable and portable at the same time.

## Why it matters

Together these four properties give you:

::card-group
  :::card{icon="i-lucide-repeat" title="Idempotent applies"}
  Re-running `up` writes nothing when nothing changed.
  :::

  :::card{icon="i-lucide-git-compare" title="Minimal diffs"}
  Only records that truly changed are rewritten.
  :::

  :::card{icon="i-lucide-globe" title="Portability"}
  The same files land valid on any shop, without churn.
  :::

  :::card{icon="i-lucide-eraser" title="Exact teardown"}
  Stable IDs mean `down` removes precisely what was created.
  :::
::

This determinism is what the [manifest](/docs/usage/concepts/the-manifest) records and compares against on every run.
