Images are different from every other entity in Shopware: a `media` row on its own is just metadata. It renders nothing until the actual file bytes are uploaded through a separate endpoint. Fakeware hides that two-step dance behind the `media` helper, and lets you attach images to a product **inline** with the `cover` and `gallery` fields — no separate `media` entity, no `product_media` / `coverId` wiring by hand.

## The `media` helper

`media(input)` describes one image and where its bytes come from.

```ts
import { media } from '@fakeware/core'

media({ url: 'https://picsum.photos/seed/hoodie/800.jpg', alt: 'Hoodie' })
media({ file: './assets/sticker.png', alt: 'Sticker' })
```

Give each image a **`url`** (Shopware downloads it server-side) **or** a **`file`** (a local path whose bytes Fakeware uploads), exactly one of the two. The file extension is read from the URL/path; pass `extension` explicitly when it can't be inferred (e.g. a URL with no suffix).

By default media lands in the shop's **Product Media** folder so the Admin groups it correctly. Override with `mediaFolderId` if you need another folder.

## Inline `cover` and `gallery`

Put a `media()` literal straight on the product. `cover` is the single main image; `gallery` is an array of extra images. Fakeware hoists each literal into a real `media` row, uploads its bytes, builds the `product_media` associations, and sets `coverId` for you — the cover comes first, then the gallery entries in order.

```ts
import { define, media } from '@fakeware/core'

define('product', {
  $key: 'hoodie',
  name: 'Hoodie',
  cover: media({ file: './assets/hoodie.png', alt: 'Hoodie' }),
  gallery: [
    media({ url: 'https://picsum.photos/seed/hoodie-back/800/800.jpg', alt: 'Back' }),
    media({ url: 'https://picsum.photos/seed/hoodie-side/800/800.jpg', alt: 'Side' }),
  ],
})
```

No `ctx`, no generator wrapping, no helper call — `cover`/`gallery` are plain fields. If you set `gallery` but omit `cover`, the first gallery entry becomes the cover.

The hoisted `media` row gets a **deterministic id derived from the owning record and slot** (e.g. `product:hoodie:cover`, `product:hoodie:0`), so re-running `up` targets the same rows and stays idempotent. Change an image's `url`/`file` and only that product and its image re-upload; leave it and nothing happens (the image is part of the product's content hash).

### Sharing one image across products

For an image reused by many products (a brand logo, a size chart), define it once as a `media` entity and reference it — `cover`/`gallery` accept a ref as well as a literal, and you can mix them:

```ts
import { define, media, ref } from '@fakeware/core'

define('media', [media({ $key: 'brand-logo', file: './assets/logo.png', alt: 'Logo' })])

define('product', {
  $key: 'hoodie',
  cover: media({ url: 'https://picsum.photos/seed/hoodie/800/800.jpg' }),
  gallery: [ref('media').key('brand-logo')],
})
```

A referenced media row is written once and shared; inline literals are written per product. Give an inline literal an explicit `$key` if you want to control its id or share it across records.

## Uploading happens automatically

On `up`, Fakeware writes the `media` rows first, then uploads each row's bytes from the URL or the local file, before the products that reference them. The file name is deterministic per record, so re-running `up` never trips Shopware's duplicate-filename check.

::note
URL upload relies on Shopware's remote-download feature being enabled (it is by default). If a shop disables it, use `file` instead.
::

## Reverting

`down` deletes the `product_media` and `media` rows in reverse order; removing a media row removes its uploaded file. No special handling: hoisted media behave like any other entity in the [manifest](/docs/usage/concepts/the-manifest).
