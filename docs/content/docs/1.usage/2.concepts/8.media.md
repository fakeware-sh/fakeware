Images are different from every other entity in Shopware: a `media` row on its own is just metadata. It renders nothing until the actual file bytes are uploaded through a separate endpoint. Fakeware hides that two-step dance behind the `media` helper, and gives products a `cover`/`gallery` shortcut so you never touch the `product_media` / `coverId` wiring by hand.

## The `media` helper

`media(input)` describes one image and where its bytes come from. It's a normal `define('media', …)` record, so everything you know about [references](/docs/usage/concepts/references) works on it.

```ts
import { define, media } from '@fakeware/core'

define('media', [
  media({ $key: 'hoodie',  url: 'https://picsum.photos/seed/hoodie/800.jpg', alt: 'Hoodie' }),
  media({ $key: 'sticker', file: './assets/sticker.png',                     alt: 'Sticker' }),
])
```

Give each image a **`url`** (Shopware downloads it server-side) **or** a **`file`** (a local path whose bytes Fakeware uploads), exactly one of the two. The file extension is read from the URL/path; pass `extension` explicitly when it can't be inferred (e.g. a URL with no suffix).

By default media lands in the shop's **Product Media** folder so the Admin groups it correctly. Override with `mediaFolderId` if you need another folder.

## Uploading happens automatically

On `up`, Fakeware writes the `media` rows, then uploads each row's bytes right after, from the URL or the local file. The file name is deterministic per record, so re-running `up` targets the same media row and never trips Shopware's duplicate-filename check. Change a `url` or `file` and only that image re-uploads; leave it and nothing happens.

::note
URL upload relies on Shopware's remote-download feature being enabled (it is by default). If a shop disables it, use `file` instead.
::

## Covers and galleries

A product's cover is **not** the media id; it's the id of a `product_media` row that links the product to the media. Getting this wrong is the classic silent bug. `cover` and `gallery` build the association and set `coverId` correctly for you:

```ts
import { cover, define, gallery, many, ref } from '@fakeware/core'

define('product', many(100, (ctx) => ({
  name: faker.commerce.productName(),
  // one image, used as the cover:
  ...cover(ref('media').pick(), ctx),
})))
```

For several images, `gallery` keeps order, numbers positions, and uses the first as the cover:

```ts
...gallery([ref('media').key('hoodie'), ref('media').at(3)], ctx)
```

Both take `ctx` (the generator's context) so the generated `product_media` ids are deterministic across runs, the same guarantee as [order builders](/docs/usage/reference/order-builders).

::warning
Give each gallery entry a **distinct** media reference. Within one record, repeated `ref('media').pick()` calls share the record seed and resolve to the *same* image, so use `.key(...)`, `.at(...)`, or a single `.pick()` for the cover.
::

## Reverting

`down` deletes the `product_media` and `media` rows in reverse order; removing a media row removes its uploaded file. No special handling: media behaves like any other entity in the [manifest](/docs/usage/concepts/the-manifest).

See the [Media helpers reference](/docs/usage/reference/media-helpers) for every option.
