Fakeware turns definition files into synced Shopware records through a fixed pipeline. Understanding that pipeline makes every other concept fall into place.

## The pipeline

A single `fakeware up` runs these stages in order:

::steps{level="3"}

### Discover

Fakeware lists your `data/` directory and collects every `.ts` file (skipping tests and declarations). This is the raw set of definition modules to evaluate.

### Evaluate

Each file is imported and executed. Its top-level `define()` calls register records into an in-memory registry. After this stage, Fakeware holds every declared record, grouped by entity, in declaration order.

### Build the write plan

Fakeware resolves the whole dataset: it assigns each record a deterministic ID, runs your generator functions, resolves `ref` and `shop` tokens to real IDs, records the dependency edges between entities, hashes each record's content, and topologically sorts everything so dependencies are written first.

### Diff against the manifest

Fakeware reads the manifest for this shop and compares each record's hash to what was committed last time. Records split into **created**, **updated** and **unchanged**.

### Sync

Entity by entity, in dependency order, Fakeware writes the created and updated records through the Shopware Sync API (skipping entities with nothing to write) and updates the manifest as it goes.

::

`fakeware down` runs the mirror image: read the manifest, delete records in reverse dependency order, converging across passes if Shopware blocks a deletion.

## A worked example

Suppose you define a tax and two products that reference it:

```ts [data/catalog.ts]
import { define, ref } from '@fakeware/core'

define('product', [
  { $key: 'a', name: 'A', taxId: ref('tax').key('standard') },
  { $key: 'b', name: 'B', taxId: ref('tax').key('standard') },
])

define('tax', { $key: 'standard', name: 'Standard', taxRate: 19 })
```

Even though `product` is declared *before* `tax`, Fakeware sees that products depend on the tax and orders the writes correctly:

```bash
up: writes tax → then product
```

The declaration order in your files never matters. Only the reference graph does.
