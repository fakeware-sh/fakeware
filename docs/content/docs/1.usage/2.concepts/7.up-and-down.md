`up` and `down` are the two halves of Fakeware's lifecycle. Both are built to be safe to interrupt and safe to re-run. This page describes what actually happens inside each.

## `up`: incremental converging sync

A run of `up` walks the [pipeline](/docs/usage/concepts/overview) and then syncs entity by entity:

::steps{level="3"}

### Resolve the plan

Discover `data/` files, evaluate them, fetch shop context, and build the write plan: deterministic IDs, resolved references, content hashes, dependency-ordered entities.

### Diff against the manifest

For each entity, compare every record's hash to the manifest and split into created / updated / unchanged.

### Write, entity by entity

In dependency order, for each entity that has anything to write: mark it pending in the manifest, write its created and updated records through the Sync API in a single call, then confirm it in the manifest. Entities with nothing to write are skipped entirely.

### Report

Emit per-entity counts (`+` created, `~` updated, `=` unchanged) and a final total of committed changes.

::

The write itself is one Sync API request per entity, an upsert of just the changed records, not one call per record.

::note
Because writes go in dependency order and the manifest is persisted after every entity, a run that stops partway still leaves a consistent ledger of what was committed. Re-running resumes cleanly.
::

### When a write fails

If the shop rejects a write, Fakeware:

1. Reports the failure with the real Shopware error: status code, entity, and the API's messages.
2. Persists a manifest containing only the entities that committed **successfully** before the failure.
3. Stops the run.

It does **not** roll back the entities already written. Keeping them in the manifest is deliberate: a later `down` can clean up the partial state, and a fixed re-run of `up` continues from where it left off.

## `down`: resilient converging revert

`down` works entirely from the manifest, with no definition files needed. It deletes recorded records in **reverse** dependency order, so referencing records go before the records they point at.

### Multiple passes

Shopware sometimes refuses to delete a record while another still references it, raising a `409`-style conflict. Fakeware treats teardown as something to *converge* rather than a single sweep:

::steps{level="3"}

### Delete what you can

Attempt each entity in reverse order. Some succeed; some raise a conflict.

### Retry the blocked ones

Entities that failed with a Shopware conflict are retried on the next pass. By now, the records that were blocking them may be gone.

### Loop until no progress

Keep passing over the remaining entities as long as each pass removes something. When a full pass makes no progress, stop.

::

If everything deletes, Fakeware removes the manifest file and the shop is clean. If some records genuinely can't be removed, those entries stay in the manifest, are reported as failures, and the command exits non-zero, so a later `down` can finish once the blocker clears.

::note
Only Shopware API conflicts are retried. An unexpected error of another kind aborts the teardown and surfaces immediately, rather than being silently swallowed.
::

## Dry runs

`up --dry-run` performs the full resolve and diff and reports what it *would* do, but makes no writes and doesn't touch the manifest. It is the safe way to validate a dataset or a config change against a real shop before committing to it.

## Failure semantics at a glance

::field-group
  :::field{name="up, write rejected"}
  Reports the real error, keeps successfully-committed entities in the manifest, stops. Re-run to resume; `down` can clean up the partial state.
  :::

  :::field{name="up, interrupted mid-entity"}
  The pending marker makes the next run re-send that entity. No duplicates, no lost state.
  :::

  :::field{name="down, delete conflict"}
  Retried across passes until it converges. Unremovable records stay in the manifest and are reported; exit code is non-zero.
  :::

  :::field{name="down, no manifest"}
  Nothing to revert, so it is a friendly no-op.
  :::
::

Every error type you might see, and how to resolve it, is catalogued in the [error reference](/docs/usage/reference/errors).
