The manifest is the single most important piece of state Fakeware keeps. It's a per-shop **ledger** of every record Fakeware has committed, and it's what makes runs incremental and teardown exact.

## What it records

After a run, Fakeware writes a manifest file into `.fakeware/`, one per shop URL. It records, for every record it committed:

- the record's **ID**
- the record's **content hash**

grouped by entity, in the order the entities were written. There's a checksum over the whole thing to detect corruption, and some bookkeeping (the Fakeware version, the shop URL, a timestamp).

```json [.fakeware/3f2a1b0c9d8e7f6a.json]
{
  "version": 2,
  "shopwareUrl": "https://my-shop.example.com",
  "entities": [
    { "entity": "tax", "records": [{ "id": "…", "hash": "…" }] },
    { "entity": "product", "records": [{ "id": "…", "hash": "…" }] }
  ],
  "checksum": "…"
}
```

::note
One manifest per shop URL. The filename is a hash of the URL, so a single project can target several shops without their state colliding.
::

## The ledger principle

The mental model to hold onto: **the manifest is a ledger you converge, not a snapshot you overwrite.** Fakeware unions records into it and reconciles them; it never optimistically throws state away.

That principle explains the behavior you'll see:

- On `up`, entities are appended as they confirm, and the manifest is rewritten after **each** entity, so an interrupted run leaves a valid, smaller ledger rather than a corrupt one.
- On `down`, only the records Fakeware actually managed to delete are removed from the ledger. A partially-blocked teardown leaves a smaller-but-valid manifest that a later `down` can finish.

## Driving incremental sync

On every `up`, Fakeware reads the prior manifest and compares each record's freshly-computed hash to the stored one:

::field-group
  :::field{name="hash matches" type="unchanged"}
  The record is skipped, with no API call.
  :::

  :::field{name="hash differs" type="updated"}
  The record is rewritten.
  :::

  :::field{name="no prior entry" type="created"}
  The record is written for the first time.
  :::
::

This is why a repeat run reports `Already up to date. Nothing changed.` and why editing one record touches only that record. The manifest is the memory that makes the diff possible.


## Driving exact teardown

`down` needs no definitions at all, since it works purely from the manifest. It reads the recorded IDs and deletes them, in **reverse** dependency order, so referencing records go before the records they reference.

Because the IDs come from the ledger (not recomputed from your current files), `down` removes exactly what was created, even if you've since changed or deleted the definition files.

## Crash safety

Manifest writes are designed to survive an interrupted run:

- Before writing an entity, Fakeware marks it **pending** in the manifest (a write-ahead flag), then writes the records, then confirms the entity. If the process dies mid-entity, the next run sees the pending marker and re-sends that entity rather than trusting it as done.
- The file itself is written atomically (to a temporary file, then renamed) so you never end up with a half-written manifest.
- A checksum guards the file; if it doesn't match on read, Fakeware refuses to trust a corrupt manifest rather than silently doing the wrong thing.

## Versioning

The manifest format is versioned. Fakeware reads exactly the version it understands and rejects newer or unknown versions with a clear error instead of guessing. This keeps upgrades safe.

## Practical rules

::warning
Treat `.fakeware/` as local state and **git-ignore it**. It's per-environment, not source.

Don't delete a manifest while its records still live in the shop, because the manifest is how `down` finds them. If you lose it, Fakeware can no longer clean those records up automatically, and you'd remove them by hand in the admin.
::

The [Up and down](/docs/usage/concepts/up-and-down) page walks through how the manifest drives a full run. The manifest reader (`readManifest`) is exported from `@fakeware/core` for advanced tooling.
