Records rarely stand alone: a product has a tax, an order has line items, a category has a parent. You express these links with `ref`, and Fakeware turns them into real IDs and a correct write order.

## `ref(entity)`

`ref('entity')` returns a builder with four ways to point at records of that entity:

::field-group
  :::field{name=".key(key)"}
  A single record by its `$key`. One-to-one, exact.
  :::

  :::field{name=".at(index)"}
  A single record by its position in the batch. Useful when records have no `$key`.
  :::

  :::field{name=".pick(count?)"}
  A seeded random selection. With no argument, one random record. With a count, that many distinct records as an association list.
  :::

  :::field{name=".all()"}
  Every record of the entity, as an association list.
  :::
::

### By key

The most common form. Point at a specific record you gave a `$key`:

```ts
define('product', {
  name: 'Trailblazer',
  taxId: ref('tax').key('standard'),
})
```

Resolving a key that doesn't exist is an error, and Fakeware tells you which entity and key failed.

### By index

Reference a record by its position in the `define` batch. This is handy for generated records that don't carry keys:

```ts
lineItems: b.lineItems.products([
  { product: ref('product').at(0), label: 'First product', unitPrice: 9.99 },
])
```

Out-of-range indexes are reported with the actual record count.

### Random picks

`pick` selects records **deterministically at random**. The choice is driven by the record's seed, so the same record always makes the same pick:

```ts
manufacturerId: ref('product_manufacturer').pick(),          // one random manufacturer
categories: ref('category').pick(3),                         // three distinct categories
```

- `pick()` returns a single ID.
- `pick(n)` returns an association list of `n` distinct records (`[{ id }, …]`), capped at the number available.

Because picks are seeded, they're stable across runs: a product keeps the same categories every time you apply.

### All records

`ref(entity).all()` expands to every record of an entity as an association list:

```ts
import { define, ref } from '@fakeware/core'

define('some_entity', {
  everyCategory: ref('category').all(),
})
```

## Association lists vs. single IDs

The form you choose matches the Shopware field you're setting:

- **To-one** fields (`taxId`, `manufacturerId`, `parentId`) take a single ID, so use `.key`, `.at`, or `.pick()`.
- **To-many** associations (`categories`, `properties`) take a list of `{ id }` objects, so use `.pick(n)` or `.all()`.

## Write ordering comes for free

When Fakeware resolves a reference, it records a dependency: "this entity needs that entity to exist first." From all these edges it builds a graph and topologically sorts the entities, so a referenced entity is always written before the entity that references it.

```ts
define('product', { taxId: ref('tax').key('standard') })
define('tax', { $key: 'standard', taxRate: 19 })
```

```bash
up: tax → product
```

You never sequence writes by hand. Declaration order and file names are irrelevant; the reference graph decides.

::warning
If your references form a **cycle** (A depends on B which depends on A), Fakeware can't pick a safe order and reports a graph error. Break the cycle by restructuring the data (for example, set one side of the relationship in a follow-up entity).
::

## Self-references and intra-entity order

An entity can reference **itself**. The classic case is a category tree, where children point at parents:

```ts
define('category', [
  { $key: 'root', name: 'Catalog' },
  { $key: 'bikes', name: 'Bikes', parentId: ref('category').key('root') },
])
```

A self-reference doesn't make the entity depend on itself as a whole. Instead, Fakeware sorts records *within* the entity so a referenced record (the parent) is written before the record that points at it (the child). Parent-before-child ordering is automatic.

## Shop references

References to *your own* records use `ref`. References to data that already exists in the shop (currencies, taxes, countries, salutations, states) use `shop`. That's the subject of the next page: [Shop context](/docs/usage/concepts/shop-context).
