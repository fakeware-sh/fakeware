Once a project is set up, your whole interaction with Fakeware is three commands: `up`, `down`, and `up --dry-run`. This page covers how to use them well.

## The core loop

```bash [Terminal]
fakeware up --dry-run   # preview what would change
fakeware up             # apply it
fakeware down           # remove everything Fakeware created
```

- **`up`** discovers your `data/` files, resolves references, and syncs the result. It writes only what changed since the last run.
- **`down`** reads the manifest and deletes exactly the records `up` created, in reverse dependency order.
- **`--dry-run`** on `up` does the full resolve and diff but writes nothing, so you can inspect the plan safely.

Full flags and behavior live in the [CLI reference](/docs/reference/cli).

## Re-running is safe

`up` is **idempotent**. Every record gets a deterministic ID and a content hash; on each run Fakeware compares hashes against the manifest and writes only the records that actually differ.

```bash [Terminal]
fakeware up
# Committed 210 changes

fakeware up
# Already up to date. Nothing changed.
```

This means you can:

- Run `up` repeatedly without creating duplicates.
- Edit one product and re-run, and only that record is rewritten.
- Add a new `data/` file and re-run, and only the new records are created.

::tip
Because IDs are derived from the entity and the record's `$key` (or its index), keep `$key` values and `ctx.index`-based fields stable. Changing a `$key` produces a *new* record rather than updating the old one. See [Determinism](/docs/concepts/determinism).
::

## Previewing before you write

Reach for `--dry-run` whenever you're unsure:

```bash [Terminal]
fakeware up --dry-run
```

It connects, fetches shop context, resolves every definition, and reports the created / updated / unchanged counts per entity, but makes no API writes and doesn't touch the manifest. It's the safe way to validate a new dataset or a config change against a real shop.

## Tearing down

`down` is resilient. Shopware sometimes refuses to delete a record while another still references it (a `409`-style conflict). Fakeware handles this by reverting in **multiple passes**: it removes what it can, retries the blocked entities once their dependents are gone, and converges.

```bash [Terminal]
fakeware down
# Removed 210 records from https://my-shop.example.com
```

If some records genuinely can't be removed (for example, an order that Shopware won't let you delete), `down` reports the failures, keeps those entries in the manifest, and exits non-zero, so a later `down` can finish the job once the blocker clears.

Skip the confirmation prompt in scripts with `--yes`:

```bash [Terminal]
fakeware down --yes
```

## Working against multiple shops

The manifest is keyed by shop **URL**, so one project can safely target several shops. Point at whichever shop you want via your `.env` (or a per-shop config with `--config`), and Fakeware keeps a separate manifest for each. Applying to staging never interferes with local.

## In scripts and CI

Fakeware is built to be non-interactive when it needs to be:

```bash [Terminal]
fakeware up
fakeware down --yes
```

A typical CI setup provisions a fresh shop (for example, a dockware container), seeds it, runs tests, and tears down:

```yaml [.github/workflows/e2e.yml]
- run: fakeware up
- run: npm test
- run: fakeware down --yes
  if: always()
```

::note
Provide credentials through the environment (`SHOPWARE_URL`, `SHOPWARE_CLIENT_ID`, `SHOPWARE_CLIENT_SECRET`) so nothing is hard-coded. Fakeware reads them the same way it reads your local `.env`.
::

## When something goes wrong

If the shop rejects a write, Fakeware surfaces the real Shopware error (status code, entity, and the messages the API returned) rather than a generic failure. An interrupted `up` leaves a valid manifest of what was already committed, so a re-run resumes cleanly and a `down` can still clean up the partial state. The [error reference](/docs/reference/errors) lists each error type and how to resolve it.
