---
title: Project structure
description: How a Fakeware project is laid out, covering the config file, the data directory, environment secrets, and the manifest directory.
---

A Fakeware project is an ordinary Node package with a small, predictable layout:

```bash
my-seed/
├── fakeware.config.ts     # Shop connection + options
├── data/                  # Your entity definitions
│   ├── tax.ts
│   ├── categories.ts
│   ├── products.ts
│   └── orders.ts
├── .env                   # Secrets (git-ignored)
├── .fakeware/             # Per-shop manifests (git-ignored)
├── package.json
└── tsconfig.json
```

## `fakeware.config.ts`

The config file is the entry point. It exports a call to `defineConfig` describing how to reach your shop:

```ts [fakeware.config.ts]
import { defineConfig } from '@fakeware/core/config'

export default defineConfig({
  shopware: {
    url: '$SHOPWARE_URL',
    clientId: '$SHOPWARE_CLIENT_ID',
    clientSecret: '$SHOPWARE_CLIENT_SECRET',
  },
})
```

Fakeware looks for `fakeware.config.ts` in the current directory and walks **up** the directory tree until it finds one, so you can run the CLI from a subfolder. The directory containing the config is the **project root**; `data/` and `.fakeware/` are resolved relative to it.

::tip
Point at a different config with `--config <path>` on any command. See the [CLI reference](/docs/reference/cli).
::

The full set of config options, including the function form for environment-aware config, is in the [Configuration reference](/docs/reference/configuration).

## `data/` directory

This is where your definitions live. Fakeware **discovers and evaluates every `.ts` file** in `data/` (recursively), except:

- `*.test.ts`: test files are skipped
- `*.d.ts`: declaration files are skipped

Each file runs its top-level `define()` calls, which register records. How you split entities across files is entirely up to you: one file per entity, one big file, or grouped by feature. The write order is determined by **references**, not by file names, so file organization is purely for your own clarity.

::note
A missing `data/` directory is not an error. Fakeware simply has nothing to apply. This is handy for a config-only project you build up gradually.
::

## `.env`

Secrets and environment-specific values live in `.env` at the project root, and are referenced from the config with the `$NAME` syntax:

```bash [.env]
SHOPWARE_URL=https://my-shop.example.com
SHOPWARE_CLIENT_ID=your-client-id
SHOPWARE_CLIENT_SECRET=your-client-secret
```

Fakeware reads `.env` itself (no extra dependency) and merges it over `process.env`. Add `.env` to `.gitignore` so credentials never get committed. See [Configuration → Environment variables](/docs/reference/configuration#environment-variables) for precedence and rules.

## `.fakeware/` directory

After the first `up`, Fakeware writes a **manifest** into `.fakeware/`. There is one manifest file per shop URL (named by a hash of the URL), recording the ID and content hash of every record it created. This is what makes runs incremental and teardown exact.

```bash
.fakeware/
└── 3f2a1b0c9d8e7f6a.json   # manifest for one shop URL
```

You should **git-ignore** `.fakeware/`, since it's local, per-environment state, not source. The scaffolder sets this up for you.

::warning
Don't hand-edit or delete manifest files while data still exists in the shop. The manifest is how `down` knows what to remove; losing it means Fakeware can no longer clean up those records automatically. See [The manifest](/docs/concepts/the-manifest) for the full model.
::

## Package scripts

Nothing special is required, but a couple of scripts keep the workflow tidy:

```json [package.json]
{
  "scripts": {
    "seed": "fakeware up",
    "seed:reset": "fakeware down && fakeware up"
  }
}
```
