## What is Fakeware?

Fakeware is a command-line tool for filling a **Shopware 6** shop with realistic demo data. You describe the data you want (products, categories, customers, orders) as plain TypeScript objects, and Fakeware syncs them into your shop over the Shopware Admin API.

```ts [data/product.ts]
import { define, ref, shop } from '@fakeware/core'

define('product', {
  $key: 'hero-bike',
  name: 'Trailblazer 5000',
  productNumber: 'SW-10001',
  stock: 42,
  taxId: ref('tax').key('standard'),
  price: [{ currencyId: shop.defaultCurrency, gross: 1299, net: 1091.6, linked: true }],
})
```

```bash [Terminal]
fakeware up
```

That is the whole loop: **define data, run `up`**. When you're done, `fakeware down` removes exactly what it created.

## The problem it solves

Populating a Shopware shop for development, demos, QA or screenshots usually means one of three bad options: clicking through the admin by hand, writing brittle SQL, or maintaining a pile of one-off import scripts. All three are slow, hard to reproduce, and messy to clean up.

Fakeware replaces them with a single source of truth (typed definition files) that are:

::card-group
  :::card{icon="i-lucide-repeat" title="Reproducible"}
  The same definitions always produce the same records, in any environment.
  :::

  :::card{icon="i-lucide-git-compare" title="Diffable"}
  Definitions live in your repo as ordinary code you can review and version.
  :::

  :::card{icon="i-lucide-eraser" title="Reversible"}
  Everything Fakeware writes is tracked, so teardown is exact and safe.
  :::

  :::card{icon="i-lucide-terminal" title="Scriptable"}
  Idempotent commands make it safe to run in setup scripts and CI.
  :::
::

## The mental model

Fakeware has a small, consistent mental model. Four ideas carry almost everything:

::steps{level="3"}

### Define entities

You call `define(entity, records)` to declare Shopware entities. A record is a plain object whose fields map to Shopware's own field names. You can generate many at once and use any data library (like Faker) inside.

### Reference other entities

Instead of hardcoding IDs, you point at other records with `ref('entity')` and at live shop data (currencies, taxes, countries) with `shop`. Fakeware resolves these to real IDs and writes everything in dependency order.

### Apply with `up`

`fakeware up` discovers your definition files, resolves everything, and syncs it to the shop. It only writes what actually changed since the last run.

### Revert with `down`

Fakeware records what it created in a per-shop **manifest**. `fakeware down` reads that manifest and removes precisely those records, nothing more.

::

## How it fits Shopware

Fakeware talks to Shopware through the **Admin API**, authenticating with an integration's client ID and secret. It writes through the Sync API, so a run is a small number of batched requests rather than hundreds of individual calls.

Because references like `shop.defaultCurrency` or `ref('tax').key('standard')` are resolved against the *live* shop at apply time, the data you author stays portable: the same files land valid against a fresh dockware container, a staging shop, or a colleague's machine, each with its own internal IDs.

::note
Fakeware never touches data it didn't create. It identifies its own records by deterministic IDs and tracks them in a manifest, so it will not overwrite or delete anything you added by hand.
::
