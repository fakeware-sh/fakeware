Everything you seed starts with `define`. It's the one function you'll use in every data file, and its shape is deliberately small.

## `define(entity, records)`

`define` registers records for a Shopware entity. The first argument is the entity's technical name; the second is a single record or an array of them.

```ts
import { define } from '@fakeware/core'

define('tax', { $key: 'standard', name: 'Standard rate', taxRate: 19 })

define('category', [
  { $key: 'root', name: 'Catalog' },
  { $key: 'bikes', name: 'Bikes', parentId: ref('category').key('root') },
])
```

A **record** is a plain object whose keys are Shopware's own field names (`name`, `taxRate`, `stock`, `price`, …). Fakeware doesn't invent a schema on top of Shopware: you're writing the same fields the Admin API expects, with references and shop lookups mixed in.

::note
The entity name is the Shopware technical name (`product`, `product_manufacturer`, `property_group_option`, …). Fakeware's typed entity map gives you autocompletion for the common ones and accepts any string for the rest.
::

## Keys with `$key`

The special `$key` field gives a record a stable, human-readable handle. It is **not** sent to Shopware, since Fakeware strips it before writing, but it does two important things:

- It's how other records reference this one: `ref('tax').key('standard')`.
- It seeds the record's deterministic ID, so the same `$key` always maps to the same Shopware record.

```ts
define('tax', { $key: 'standard', name: 'Standard rate', taxRate: 19 })
//                ^^^^^^^^^^^^^^^^ referenced elsewhere as ref('tax').key('standard')
```

::warning
`$key` values must be unique within an entity. Defining two records with the same `$key` for the same entity is an error. Changing a `$key` later produces a **new** record rather than updating the existing one.
::

Records without a `$key` are still valid; they simply can't be referenced by key. They get IDs from their position instead (see [Determinism](/docs/concepts/determinism)).

## Generating many with `many`

Most seed data is bulk data. `many(n, fn)` produces `n` records from a generator function:

```ts
import { define, many } from '@fakeware/core'

define(
  'product',
  many(100, (ctx) => ({
    name: `Product ${ctx.index + 1}`,
    productNumber: `SW-${10000 + ctx.index}`,
    stock: 100,
  })),
)
```

The generator receives a **context** object, `ctx`, for the record it's producing:

::field-group
  :::field{name="ctx.index" type="number"}
  The zero-based position of this record within the batch. Use it for unique, stable fields like product numbers.
  :::

  :::field{name="ctx.count" type="number"}
  The total number of records in the batch.
  :::

  :::field{name="ctx.seed" type="number"}
  A deterministic seed derived from the entity and index. It is used internally to make random picks stable, and it is available if you want your own stable randomness.
  :::

  :::field{name="ctx.shop" type="Shop"}
  Access to the live shop context for advanced lookups (the same data behind the `shop` object).
  :::
::

::tip
Use `ctx.index` for anything that must stay stable across runs. A field derived from `ctx.index` produces the same value every time, so re-running `up` sees no change. A field derived from unseeded randomness (a bare `Math.random()`) changes every run and will churn.
::

### Generators run at apply time

The function you pass to `many` (or a function used as a whole record) isn't called when the file is evaluated. It runs later, during the build-plan stage, once the shop context is available. That's why `shop` lookups and references work inside it.

## Records as functions

A record can itself be a function of `ctx`, not just an object. This is what `many` produces under the hood, and you can use it directly for a single generated record:

```ts
define('product', (ctx) => ({
  name: `The one product`,
  productNumber: `SW-${10000 + ctx.index}`,
}))
```

## Keying a generated list

When you build a list yourself and want each item referenceable, `keyed` attaches a `$key` to every item from a function, de-duplicating collisions automatically:

```ts
import { define, keyed } from '@fakeware/core'

const colors = ['Red', 'Green', 'Blue']

define(
  'property_group_option',
  keyed(colors.map((name) => ({ name })), (item) => item.name.toLowerCase()),
)
```

Each option becomes referenceable as `ref('property_group_option').key('red')`, and if two items would produce the same key, `keyed` appends a numeric suffix so keys stay unique.
