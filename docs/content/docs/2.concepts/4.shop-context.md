---
title: Shop context
description: How Fakeware reads currencies, taxes, countries and states from the live shop, and how the shop object resolves them to real IDs.
---

Some IDs don't belong to your definitions; they already exist in the shop. The default currency, the standard tax, the country "Germany", the "open" order state: these are fixed by the store, and differ between installations. The `shop` object lets you reference them by meaning, and Fakeware resolves them to that shop's real IDs at apply time.

## Why it exists

If you hardcoded a currency UUID, your data would only work against one specific shop. Write `shop.defaultCurrency` instead, and the same definitions land valid against any shop (local, staging, or a fresh dockware container), each with its own internal IDs.

```ts
price: [{ currencyId: shop.defaultCurrency, gross: 19.99, net: 16.8, linked: true }]
```

## What Fakeware fetches

At the start of a run, Fakeware queries the shop once and builds a **shop context**, an indexed snapshot of the lookup data it needs:

::field-group
  :::field{name="currencies"}
  By ISO code, plus the system default.
  :::

  :::field{name="languages"}
  By locale, plus the default.
  :::

  :::field{name="sales channels"}
  By name, plus the default (first active).
  :::

  :::field{name="countries"}
  By ISO code.
  :::

  :::field{name="salutations"}
  By salutation key.
  :::

  :::field{name="state machine states"}
  By state machine and technical name.
  :::

  :::field{name="taxes"}
  By rate (default is the highest rate).
  :::

  :::field{name="payment &amp; shipping methods"}
  By technical name.
  :::
::

## The `shop` object

`shop` exposes two kinds of members: **defaults** (no argument) and **lookups** (by key).

### Defaults

```ts
shop.defaultCurrency
shop.defaultLanguage
shop.defaultSalesChannel
shop.defaultCountry
shop.defaultSalutation
shop.defaultTax
shop.defaultPaymentMethod
shop.defaultShippingMethod
```

### Lookups

```ts
shop.currency('EUR')
shop.language('en-GB')
shop.salesChannel('Storefront')
shop.country('DE')
shop.salutation('mr')
shop.tax(19)
shop.paymentMethod('payment_invoice')
shop.shippingMethod('standard')
```

### State machine states

Orders and their deliveries and transactions each move through a state machine. There are convenience lookups for the three common ones, plus a general form:

```ts
shop.orderState('open')
shop.orderDeliveryState('shipped')
shop.orderTransactionState('paid')
shop.stateMachineState('order.state', 'completed')
```

## How lookups resolve

Each `shop.*` call returns a **token**, not an ID. The token carries a descriptor (like `currency:EUR`) and the logic to resolve it. During the build-plan stage, Fakeware resolves every token against the fetched context and substitutes the real ID.

This deferral is what lets you write shop lookups anywhere in a record or generator. They only need the context, which is ready by the time resolution happens.

::note
Shop tokens don't create dependency edges the way `ref` does. They point at data that already exists in the shop, so there's nothing for Fakeware to write first.
::

## Helpful errors

If a lookup doesn't match, Fakeware throws a clear error listing what *is* available:

```text
country 'XX' not found. Available: DE, AT, CH, GB, US.
```

For states, it scopes the hint to the relevant state machine:

```text
stateMachineState('order.state', 'shpped') not found.
States for 'order.state': open, in_progress, completed, cancelled.
```

This turns a typo into a one-line fix instead of a mysterious API rejection.

## Reaching the raw context

For advanced cases you can read the resolved context directly inside a generator via `shop.context()`, and plugin-provided data via `shop.extensions`. These are only available while definitions are being resolved. Most data never needs them, since the tokens above cover the common ground.

The full list of lookups and context types is in the [Shopware API reference](/docs/reference/shopware-api).
